Design & Architecture
Overview

The architecture of smART follows a decentralized model that integrates on-chain logic, event indexing, distributed storage, and a modern Web3 interface. Its goal is to minimize intermediaries in freelancer-client interactions and ensure transparency, integrity, and impartial dispute resolution.

The platform is structured into three layers: a blockchain layer that executes the core business logic, an off-chain event indexing and processing layer, and a frontend DApp layer that enables users to interact seamlessly with the system. It also includes a GenAI validation service backed by PostgreSQL and Redis for caching and fast retrieval.

Main Components
  • Smart Contracts: Handle state transitions, payments, deadlines, and contract lifecycle.
  • Kleros: Decentralized arbitration system used for resolving disputes through juror voting.
  • Ponder: Framework for indexing on-chain events into a relational PostgreSQL database.
  • IPFS: Distributed storage for large deliverables and dispute evidence.
  • GenAI Validation Service: Multimodal model for deliverable verification, backed by PostgreSQL and Redis.
  • Frontend Web3: DApp built with Scaffold-ETH 2 and deployed on Vercel.
On-chain Layer

The blockchain layer manages the lifecycle of each work agreement, ensuring deterministic and verifiable execution. Smart Contracts enforce deadlines, freeze and release funds, and record immutable references to off-chain assets. Their internal logic is detailed in the next section.

Decentralized Storage

Large audiovisual deliverables and dispute evidence are stored in IPFS. IPFS (InterPlanetary File System) is a peer-to-peer distributed file system that seeks to connect all computing devices with the same system of files. Uploaded files are identified by their content hash. Only the resulting hash is written on-chain, allowing the blockchain to act as an integrity verifier while avoiding the cost of storing large files directly on the network.

Decentralized Dispute Resolution

Disputes between freelancers and clients are resolved through Kleros, which assigns a random set of jurors who analyze the evidence and vote on the outcome. The system includes fee deposits, reimbursements, and an appeal mechanism with additional staking incentives to discourage frivolous claims.

Event Indexing Layer

Since several application views require aggregated or historical information, Ponder listens to blockchain events and synchronizes them into a PostgreSQL database. This enables efficient querying of listings, contract histories, user activity, and dispute states.

GenAI Processing Layer

A multimodal AI model analyses submitted deliverables to verify whether they align with the work agreed upon. The results are cached through two layers: PostgreSQL for persistence and Redis for fast retrieval, reducing repetitive inference and improving performance.

Frontend Architecture

The user interface is built on Scaffold-ETH 2, which provides wallet connectivity, contract interaction hooks, and reusable Web3 components. The final DApp is deployed on Vercel as a Platform-as-a-Service (PaaS), enabling fast iteration and continuous delivery.
